# @mytvtime/db

Drizzle ORM schema, typed client factory, and versioned SQL migrations for
**MyTVTime** on **PostgreSQL 16**. This package is the single source of truth for
the database contract — every other package (`apps/web`, `packages/importer`)
imports its schema and inferred types from here and never reaches for raw SQL DDL.

## Install / build

```bash
pnpm install
pnpm build      # tsc → dist/ (ESM + .d.ts)
```

## Usage

```ts
import { createDb, catalogShows, type Show, type NewShow } from '@mytvtime/db';

const { db, client } = createDb(process.env.DATABASE_URL!);

const shows: Show[] = await db.select().from(catalogShows);

await db.insert(catalogShows).values({ tmdbId: 1399, name: 'Game of Thrones' });

await client.end(); // close the pool on shutdown
```

- `createDb(url, opts?)` → `{ db, client }`. `db` is the typed Drizzle instance
  (relational `db.query.*` helpers included); `client` is the underlying
  postgres.js connection — call `client.end()` to close.
- The full `schema` object and every table are re-exported (via `export *`), so
  `import * as schema from '@mytvtime/db'` or `'@mytvtime/db/schema'` both work.
- Every table exposes inferred types: `Show`/`NewShow`, `Episode`/`NewEpisode`,
  `Follow`/`NewFollow`, … plus enum unions `FollowStatus`, `ListKind`, `RefType`.

## Schema

Defined in [`src/schema.ts`](./src/schema.ts). 17 tables split into a shared
**catalogue** (TMDB cache: `catalog_shows`, `catalog_seasons`, `catalog_episodes`,
`catalog_movies`) and **user data** (`users`, `settings`, `follows`,
`episode_watches`, `movie_watches`, `episode_ratings`, `movie_ratings`,
`episode_emotions`, `lists`, `list_items`, `comments`, `stats_monthly`,
`user_stats`).

Three Postgres enum types: `follow_status`, `list_type`, `ref_type`.
All ids are `bigint` (`bigserial` for auto-increment PKs); all timestamps are
`timestamptz`. Natural unique keys (e.g. `follows(user_id, show_id)`,
`catalog_episodes(show_id, season_number, episode_number)`) make the importer's
upserts idempotent.

## Migrations

Migrations live in [`drizzle/`](./drizzle) and are generated by drizzle-kit from
`src/schema.ts` (`pnpm generate`). Alongside each `NNNN_*.sql` file, drizzle-kit
maintains `meta/_journal.json` and per-migration snapshots (`meta/NNNN_snapshot.json`)
so the next `generate` diffs against the real baseline — keep the whole `drizzle/`
folder committed and shipped (`files: ["drizzle"]`). Referential actions:
user-data → catalog FKs use `ON DELETE RESTRICT` (never cascade away watch history);
`user_id → users` and within-catalog FKs use `ON DELETE CASCADE`.

Apply migrations one of two ways:

```bash
# Option A — runtime migrator (production-safe; used by the compose `migrate` one-shot).
# Uses drizzle-orm's postgres-js migrator (NOT drizzle-kit, which is pruned from the
# prod image). Idempotent — tracked in drizzle.__drizzle_migrations. Requires a build first.
pnpm build
DATABASE_URL=postgres://mytvtime:mytvtime@db:5432/mytvtime pnpm run migrate

# Option B — plain psql (no Node needed; ad-hoc / debugging)
psql "$DATABASE_URL" -f drizzle/0000_init.sql
```

The `migrate` script runs `node dist/migrate.js` (`src/migrate.ts`), which reads
`DATABASE_URL` from the environment and applies every migration listed in
`drizzle/meta/_journal.json`.

To evolve the schema: edit `src/schema.ts`, then regenerate SQL with
`pnpm generate` (drizzle-kit diffs the schema and writes the next
`NNNN_*.sql` + updated meta snapshot). Never hand-edit an already-applied
migration; add a new one.

> Note: `0000_init.sql` is drizzle-kit-generated and validated against a real
> `postgres:16` container (applies cleanly, idempotent re-run, and a follow-up
> `pnpm generate` reports no schema changes). The `--> statement-breakpoint`
> markers are a drizzle-kit convention; `psql` ignores them as SQL comments, so
> the file is also runnable ad-hoc via `psql -f`.
